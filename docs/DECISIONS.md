## 1. Архитектура (без изменений к ТЗ)
- Один Vite/React-build, **два режима**: браузер (Telegram Login Widget) + Telegram Mini App (`initData`). Разница только в слое авторизации (`AuthProvider`).
- Бэкенд: **FastAPI + SQLAlchemy 2.0 async + Alembic**, Pydantic v2.
- Бот: **aiogram 3.x**, на проде **webhook**.
- БД **PostgreSQL**, **Redis** (кэш). Файлы — S3-совместимое хранилище.
- Деплой: docker-compose (`frontend` Nginx-статика, `backend` Uvicorn, `bot`, `postgres`, `redis`).
- Прод: **Timeweb (РФ)** — данные ПДн в России (152-ФЗ).

## 2. Решения заказчика

| Тема | Решение |
|---|---|
| **CRM / админка** | Отдельный роут `/admin110220` в том же фронте, ленивый чанк. Путь намеренно «секретный», но это не защита. |
| **Вход в админку** | Логин + пароль + **2FA** (TOTP или код в Telegram). |
| **Напоминания** | Таблица `Reminder` + воркер раз в минуту (скан `status='scheduled' AND send_at<=now()`). НЕ per-job APScheduler. Идемпотентность через `sent_at`. |
| **Уведомления клиенту** | Подтверждение сразу + за 24ч + за 2ч + запрос отзыва после визита. |
| **Отмена / перенос** | Не позднее **6 часов** до визита. Значение в конфиге (`CANCELLATION_HOURS=6`). |
| **Исходы записи** | Мастер вручную в CRM: «прошла» → `completed`, «отменилась» → `cancelled`, «не пришёл» → `no_show`. Кнопки исхода — в один тап в карточке/календаре. |
| **Личный кабинет клиента** | Да, сразу: свои записи (ближайшие/прошлые), перенос/отмена с правилом 6ч. |
| **Мультиуслуга** | Да — несколько услуг в одной записи, длительность и цена **суммируются**. |
| **Буфер между записями** | На уровне услуги (поле `buffer_min` в `Service`), влияет на расчёт слотов. |
| **Отзывы** | Сбор через бота после визита → модерация мастером → публикация на сайте. |
| **Оплата** | Нет в первом релизе. Поле `deposit_amount` оставить, ЮKassa — позже. |
| **Язык** | Только русский, без i18n-обвязки. |
| **SEO** | Не приоритет (открывают по прямой ссылке / в боте). Только базовые мета + OpenGraph для красивой ссылки. Пререндер/SSG — нет. |
| **Двойное бронирование** | Postgres `EXCLUDE` constraint (`btree_gist` + `tstzrange`) на пересечение интервалов одного мастера. |
| **Мультимастер** | Один мастер сейчас. Модель с `master_id` везде — на вырост. UI под одного. |
| **Аналитика** | Нужно всё: выручка, кол-во записей, % отмен/no-show, заполняемость, топ услуг, новые vs постоянные. |

## 3. Контент и наполнение
- **Услуги, портфолио «до/после», график работы, отпуска/выходные (TimeOff)** — мастер заполняет сам через CRM. Значит CRUD услуг/графика нужен рано; витрина тянет из БД.
- На старте разработки — **демо-сид** (пара услуг + график), чтобы тестировать запись до готовой админки.
- Шаг сетки слотов мастер выставляет сам при построении графика.
- Текста «Обо мне» нет — сгенерировать черновик-заглушку.
- **Адрес студии — приватный**: хранится в настройках мастера, отдаётся клиенту только ПОСЛЕ записи (подтверждение бота / ЛК). На публичном сайте не показывать.
- Соцсети: Instagram (вставить аккуратно — Meta запрещена в РФ), позже Telegram-канал.
- Логотип — **буквенный/текстовый ASKBROWS** (типографика, не картинка).
- Приветствие бота `/start` — дефолтный текст.

## 5. Изменения к модели данных ТЗ (раздел 5)
Вытекают из решений выше:
- **`Appointment` ↔ `Service` — many-to-many** (мультиуслуга), а не один `service_id`. Нужна связующая таблица `appointment_services`.
- **`Service`**: добавить `buffer_min`.
- Новая модель **`Review`**: `id, client_id, appointment_id, rating, text, status(pending|approved|rejected), is_published, created_at`.
- **`Master`/настройки**: приватный `address`, `slot_step_min`, `cancellation_hours`.
- **`Client`**: факт согласия на ПДн — `consent_at` (nullable).
- Остальное по ТЗ: `Client`, `WorkingHours`, `TimeOff`, `Reminder` (типы: confirmation|day_before|hours_before|review_request).
- Все времена в БД — **UTC**, отображение `Europe/Moscow`.

## 7. Тех-решения Фазы 1 (Сессия 2, 2026-07-14)

| Тема | Решение |
|---|---|
| **Лейаут репо** | Монорепо `/frontend` + `/backend`. Бот и воркер напоминаний — **отдельные процессы из образа `backend`** (общие SQLAlchemy-модели без дублирования). В compose это разные сервисы (`backend`/`bot`/`worker`) с разными `command`. |
| **Глубина архитектуры** | Прагматичный минимум в духе PROJECT_BIBLE: заводим модули под Фазу 1 (`service`, `client`, `settings`), схему БД — целиком, остальные модули/engine-слои добавляем по фазам. Без пустых заготовок наперёд. |
| **Frontend-тулинг** | **pnpm**. Vite + React 18 + TS + Tailwind. Fonts — **self-host** (`.woff2` в `src/shared/fonts/`, пока системные фолбэки). |
| **Backend-тулинг** | **uv + ruff**, Python 3.12+. |
| **Frontend в docker-compose** | Dev — Vite dev-сервер с HMR (`Dockerfile.dev`); prod — статика за Nginx (`Dockerfile` + `nginx.conf`). |
| **API-контур** | Публичные роуты под `/api`; в dev Vite проксирует `/api` на `backend:8000`, в prod — Nginx. `GET /api/services` + `GET /api/health` уже есть; остальное каркасом. |
| **Схема БД** | Первая миграция `0001_init` (hand-written): все таблицы + `btree_gist` + EXCLUDE-констрейнт `appointment_no_overlap` (нет пересечений активных записей мастера). Enum'ы — нативные PG-типы. |

## 8. Деплой как под-сайт awwwdde-panel (Сессия 4, 2026-07-14)

| Тема | Решение |
|---|---|
| **Формат деплоя** | Единый Docker-образ по контракту awwwdde-panel: **корневой `Dockerfile`**, порт **8080**, `GET /healthz`, БД из `DATABASE_URL`, миграции+сид на старте. Панель собирает образ из репозитория `origin` сама (у заказчика нет Docker Desktop). |
| **Раздача фронта** | В образе фронт собирается в статику и раздаётся **тем же FastAPI** (`STATIC_DIR`), фолбэк на `index.html` для клиентского роутинга. Фронт и API — один origin → CORS не нужен. Мульти-контейнерный `docker-compose.yml` остаётся для локальной разработки. |
| **DATABASE_URL** | Нормализуется к `asyncpg` (панель может выдать `postgres://`/`postgresql://`/`+psycopg2`). **Только PostgreSQL** — SQLite не поддерживается (EXCLUDE/btree_gist/ARRAY/enum). |
| **Бот/воркер** | В образ панели НЕ входят (отдельные процессы). В Фазе 3 бот-webhook встроим в тот же FastAPI, воркер — фоновой задачей/отдельным сервисом. |
| **Env панели** | Минимум: `DATABASE_URL`, `JWT_SECRET`, `BOT_TOKEN`. Подробности — `DEPLOY-PANEL.md`. |

## 9. ⚠️ Открытые риски / что нужно от заказчика
1. **ПДн без оператора.** Нет ни ИП, ни самозанятости. Технически закладываем чекбокс согласия + хранение факта + шаблон политики конфиденциальности с плейсхолдерами реквизитов. **До публичного запуска** реквизиты оператора (по факту — Ангелина как физлицо) нужно вписать. Юридический риск сбора ПДн без оформленного оператора.
2. **Токен/username бота** — ждём от заказчика.
3. **Миграция из DIKIDI** — доступа к админке DIKIDI нет, экспорт под вопросом. Автоимпорт не закладываем; при наличии выгрузки — разовый скрипт (фаза 5).
4. **Реквизиты домена askbrows.ru** — когда купят, переключить в `.env`.
