## 1. Цель и ключевая бизнес-логика

Единая система из трёх связанных частей:
1. **Публичный сайт-визитка** (`askbrows.ru`) с витриной услуг и онлайн-записью.
2. **CRM** для мастера — база клиентов, расписание, история, аналитика, рассылки.
3. **Telegram-бот** (токен уже есть) — авторизация, сбор номера телефона, автоматические уведомления.

**Главная архитектурная установка (из требований заказчика):**
- Бот **уже существует**, токен есть. Сейчас в боте кнопка ведёт на DIKIDI — её надо **заменить на свой сайт**, открывающийся как **Telegram Mini App** прямо внутри Telegram.
- Клиент попадает на запись **двумя путями**: (а) обычный сайт в браузере, (б) Mini App внутри Telegram через кнопку бота.
- Авторизация — через Telegram. В браузере — Telegram Login Widget; в Mini App — автоматически через `initData`.
- Бот **забирает номер телефона** клиента (`request_contact` / `requestContact`).
- После записи данные **автоматически появляются в CRM**.
- Бот **сам шлёт уведомления** (подтверждение, напоминания).

## 2. Архитектурное решение: один фронт, две точки входа

**Решение: единый React-проект, два режима работы** (не два отдельных приложения — это убирает дублирование кода записи и услуг).

Приложение при загрузке определяет режим:
```ts
const tg = window.Telegram?.WebApp;
const isMiniApp = Boolean(tg?.initData && tg.initData.length > 0);
```
- `isMiniApp === true` → режим **Mini App**: читаем `tg.initDataUnsafe.user`, авторизация по `initData` (бэкенд валидирует подпись), применяем тему Telegram, `tg.expand()`.
- `isMiniApp === false` → режим **браузер**: авторизация через Telegram Login Widget.

Витрина, услуги, календарь слотов, личный кабинет — **общий код**. Различается только слой авторизации (вынести в `AuthProvider` с двумя стратегиями).

**Номер телефона — важный нюанс:** `initData` Mini App **не содержит телефон** (только Telegram ID, имя, username). Телефон получаем одним из путей с фолбэком:
1. В Mini App вызвать `tg.requestContact()` (поддерживается в новых версиях Telegram) → колбэк с номером.
2. Фолбэк: бот отправляет в чат кнопку `request_contact` (`KeyboardButton(request_contact=True)`), ловит `message.contact`, сохраняет телефон, связывает с клиентом.

## 3. Технологический стек

### Frontend
| Что | Технология |
|-----|-----------|
| Сборка | **Vite** |
| Фреймворк | **React 18 + TypeScript** |
| Стили | **Tailwind CSS** |
| Анимации | **Framer Motion** (UI-переходы, появления), **GSAP** (сложные таймлайны, ScrollTrigger) |
| Плавный скролл | **Lenis** (синхронизировать с GSAP ScrollTrigger через `lenis.on('scroll', ScrollTrigger.update)`) |
| Роутинг | React Router |
| Запросы | TanStack Query (React Query) + axios |
| Формы | React Hook Form + Zod |
| Telegram | `@telegram-apps/sdk` (Mini App), Telegram Login Widget (браузер) |

### Backend (Python)
| Что | Технология |
|-----|-----------|
| Фреймворк | **FastAPI** |
| ASGI-сервер | Uvicorn (за Nginx) |
| ORM | **SQLAlchemy 2.0** (async) + Alembic (миграции) |
| БД | **PostgreSQL** |
| Валидация | Pydantic v2 |
| Бот | **aiogram 3.x** |
| Очередь/планировщик | **APScheduler** (на старте достаточно) или **Celery + Redis** (если рассылок будет много) |
| Кэш/брокер | Redis |
| Auth | Валидация Telegram `initData` (HMAC-SHA256) + JWT в httpOnly cookie |
| Файлы | S3-совместимое (Selectel / Yandex Object Storage) |

### Деплой
Docker Compose: `frontend` (Nginx со статикой Vite-сборки), `backend` (Uvicorn), `bot` (aiogram, отдельный процесс/контейнер), `postgres`, `redis`. Reverse-proxy Nginx, HTTPS (обязателен для Mini App — Telegram требует валидный SSL).

> Бот и API могут жить в одном репозитории (моно-репо), но запускаться разными процессами. Бот работает на **webhook** (а не polling) на проде, чтобы FastAPI и бот делили один домен.

## 4. Архитектура

```
                          ┌──────────────────────────┐
   Браузер ──────────────▶│  Frontend (React+Vite)    │
   (Login Widget)         │  один build, 2 режима     │
                          └────────────┬─────────────┘
   Telegram ─── Mini App ──────────────┤ (initData)
   (кнопка бота)                       │ REST/JSON
                                       ▼
                          ┌──────────────────────────┐      ┌─────────────┐
                          │   FastAPI (backend)       │─────▶│ PostgreSQL  │
                          │   API записи/CRM/услуг     │      │ (SQLAlchemy)│
                          └──────┬──────────────┬──────┘      └──────┬──────┘
                                 │              │ ставит задачи      │
                                 │ webhook      ▼                    │
                                 │       ┌────────────┐              │
                                 ▼       │ APScheduler │ читает      │
                          ┌──────────────┤  / Celery   │◀────────────┘
                          │ Bot (aiogram)│  напоминания │
                          │  уведомления │└────────────┘
                          └──────┬───────┘      ▲
                                 ▼              │ Redis (брокер/кэш)
                            Telegram API        │
```

**Поток «запись → уведомление»:**
1. Клиент входит (браузер: Login Widget / Mini App: `initData`) → бэкенд валидирует, создаёт/находит `Client`.
2. Нет телефона → запрос контакта (Mini App `requestContact` или бот `request_contact`).
3. Выбор услуги + свободного слота → `POST /appointments` → `Appointment(status=pending|confirmed)`.
4. Бэкенд планирует напоминания (APScheduler jobs: подтверждение сейчас, за 24ч, за 2ч).
5. В нужный момент планировщик дёргает бота → бот шлёт сообщение.
6. Запись сразу в CRM.

## 5. Модель данных (SQLAlchemy / концептуально)

```python
# Master — мастер (заложить на будущее мультимастер)
Master:        id, name, phone, timezone="Europe/Moscow", created_at

# Client — клиент
Client:        id, telegram_id (unique), first_name, last_name, username,
               phone (заполняет бот через request_contact), photo_url,
               notes (приватные заметки мастера), tags, is_blocked, created_at

# Service — услуга
Service:       id, master_id(FK), title, description, duration_min,
               price, deposit_amount(nullable), category, photo_url, is_active

# Appointment — запись
Appointment:   id, master_id(FK), client_id(FK), service_id(FK),
               start_at(UTC), end_at(UTC),
               status: pending|confirmed|completed|cancelled|no_show,
               source: site|miniapp|bot|manual,
               comment, created_at, updated_at

# WorkingHours — график работы
WorkingHours:  id, master_id(FK), weekday(0-6), start_time, end_time, is_day_off

# TimeOff — отпуска/перерывы/выходные вне графика
TimeOff:       id, master_id(FK), start_at, end_at, reason

# Reminder — запланированное уведомление
Reminder:      id, appointment_id(FK),
               type: confirmation|day_before|hours_before|review_request,
               send_at, sent_at(nullable),
               status: scheduled|sent|failed
```

Индексы: `Appointment(master_id, start_at)`, `Client(telegram_id)`, `Reminder(status, send_at)`. Все времена в БД — **UTC**, отображение в `Europe/Moscow`.

## 7. Нефункциональные требования
- **Telegram-безопасность:** обязательная валидация подписи `initData` (HMAC-SHA256 на основе bot token) на бэкенде — без неё нельзя доверять данным фронта. Аналогично валидировать `hash` Login Widget. JWT в httpOnly+Secure cookie.
- **HTTPS обязателен** (Mini App не работает без валидного SSL).
- **152-ФЗ (ПДн):** телефоны и имена — персональные данные. Нужны: чекбокс согласия при записи + хранение факта согласия, политика конфиденциальности, удаление данных по запросу, хостинг данных в РФ.
- **Защита записи:** rate-limiting, идемпотентность создания записи, антиспам-бронирование.
- **Уведомления:** идемпотентность (`Reminder.sent_at` — не слать дважды), ретраи при ошибке Telegram API, логирование.
- **Часовые пояса:** UTC в БД, отображение `Europe/Moscow`.
- **Бэкапы:** ежедневный дамп Postgres.

**Фаза 5 — Аналитика, рассылки, миграция, полировка (1–2 нед):** дашборд, сегментные рассылки, перенос данных из DIKIDI, SEO/пререндер, оптимизация, согласие на ПДн.
